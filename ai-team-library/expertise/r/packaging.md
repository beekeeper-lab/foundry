# R Packaging

Guidelines for managing R dependencies, structuring packages, and shipping
releases. `renv` is the project-local lockfile manager that has effectively
replaced `packrat`; it is the default for both packages and analysis projects.

---

## Defaults

| Concern              | Default Tool / Approach                  |
|----------------------|------------------------------------------|
| Dependency manager   | `renv`                                   |
| Package source       | CRAN (primary), Bioconductor (genomics), GitHub via `remotes` (development) |
| Lockfile             | `renv.lock` (committed, always)          |
| Project metadata     | `DESCRIPTION` (single source of truth)   |
| Namespace            | `NAMESPACE` (generated by `roxygen2`, committed) |
| Build / test loop    | `devtools` (`load_all`, `test`, `check`) |
| Package documentation site | `pkgdown`                          |
| Release flow         | `usethis` + `devtools` + `pkgdown`       |
| Distribution         | CRAN for public packages; internal R-universe / posit-package-manager for private |

### Alternatives

- **`packrat`** — the predecessor to `renv`; deprecated. Migrate to `renv`.
- **`pak`** — a faster installer that replaces `install.packages()` and the
  `remotes` package. Use it under the hood for speed; `renv` interoperates
  with it cleanly.
- **`rspm` / Posit Package Manager** — a CRAN mirror with binary builds for
  faster installs in CI and on Linux. Configure as the `renv` source if your
  org has access.
- **`drat`** — a lightweight private-repository tool. Use only when neither
  R-universe nor PPM is available.

## DESCRIPTION Reference

```
Package: orderpricing
Type: Package
Title: Tier-Based Order Pricing Calculations
Version: 0.1.0
Authors@R:
    person("Team", "Name", email = "team@example.com", role = c("aut", "cre"))
Description: Calculates discount tiers and final order totals using a
    tidyverse-friendly tibble interface.
License: MIT + file LICENSE
URL: https://github.com/example/orderpricing
BugReports: https://github.com/example/orderpricing/issues
Depends:
    R (>= 4.3)
Imports:
    dplyr (>= 1.1.0),
    rlang (>= 1.1.0),
    tibble (>= 3.2.0)
Suggests:
    testthat (>= 3.0.0),
    covr,
    knitr,
    rmarkdown
Encoding: UTF-8
LazyData: true
RoxygenNote: 7.3.1
Config/testthat/edition: 3
```

**Key fields:**

- `Depends: R (>= ...)` pins the minimum R version.
- `Imports:` are required at runtime and installed automatically.
- `Suggests:` are optional (vignettes, tests). Code in `R/` must guard them
  with `requireNamespace("pkg", quietly = TRUE)`.
- `Config/testthat/edition: 3` opts into testthat 3rd-edition behaviour.

## Dependency Management with renv

```r
# Initialise renv in an existing project
renv::init()

# Add a dependency: install + record in renv.lock
install.packages("dplyr")
renv::snapshot()

# Restore an environment from renv.lock (CI, new clone, new machine)
renv::restore()

# Update a dependency
renv::update("dplyr")
renv::snapshot()

# Inspect what is out of sync
renv::status()
```

### What gets committed

| File / dir         | Commit? | Notes                                        |
|--------------------|---------|----------------------------------------------|
| `renv.lock`        | Yes     | The lockfile — pin record of every package   |
| `renv/activate.R`  | Yes     | Bootstraps renv when R starts                |
| `renv/settings.dcf`| Yes     | Project-level renv settings                  |
| `renv/library/`    | No      | Local installed packages — `.gitignore` it   |
| `renv/staging/`    | No      | Temporary install directory                  |

The `renv/` directory has a generated `.gitignore` that handles this — do
not edit it. Commit `renv.lock` on every dependency change with a message
that names the package and version delta.

### CRAN, Bioconductor, and GitHub installs

```r
# CRAN — the default
install.packages("dplyr")

# Bioconductor (genomics, computational biology)
install.packages("BiocManager")
BiocManager::install("DESeq2")

# GitHub development versions via remotes
install.packages("remotes")
remotes::install_github("tidyverse/dplyr")
remotes::install_github("tidyverse/dplyr@v1.1.0")  # pin to a tag
remotes::install_github("tidyverse/dplyr#1234")     # install a PR

# Local source tree
remotes::install_local("/path/to/package")
```

`renv::snapshot()` records the source for every package — CRAN, Bioconductor,
GitHub, or local — so `renv::restore()` reproduces the exact provenance.

## Build and Test Loop with devtools

```r
# Interactive development cycle (run from package root)
devtools::load_all()    # Source R/, attach exports — like `import` reload
devtools::document()    # Regenerate man/ + NAMESPACE from roxygen2 tags
devtools::test()        # Run testthat suite
devtools::check()       # Full R CMD check (CRAN-equivalent)
devtools::install()     # Install into the user library

# CI gate (one-shot)
R CMD check --as-cran .
```

`devtools::check()` is the gate that must pass before a release. It runs
`R CMD check`, which catches missing imports, undocumented exports, broken
examples, and a long list of CRAN policy violations.

## Documentation Site with pkgdown

```r
# Initialise once
usethis::use_pkgdown()

# Build the site locally
pkgdown::build_site()

# Deploy to GitHub Pages (one-time setup)
usethis::use_pkgdown_github_pages()
```

`pkgdown` reads roxygen2 docs, vignettes, and the README to produce a
documentation site. `_pkgdown.yml` controls navigation, reference grouping,
and theme. Build the site in CI on every release tag and deploy to GitHub
Pages or your internal docs host.

## Versioning

Use semantic versioning (`MAJOR.MINOR.PATCH`):

- **MAJOR** — breaking API change (renamed/removed exports, behaviour change).
- **MINOR** — new exported functions or arguments, backward compatible.
- **PATCH** — bug fixes, internal refactors.

```r
# Bump the version (edits DESCRIPTION + NEWS.md)
usethis::use_version("minor")

# Edit NEWS.md to summarise changes
usethis::edit_file("NEWS.md")
```

Maintain a `NEWS.md` file at the package root. Each version gets a section
with bullets under `# Major changes`, `# New features`, `# Bug fixes`. CRAN
maintainers and downstream users read this on every update.

## Release Flow

```r
# 1. Run release checks
devtools::check()                     # full R CMD check
devtools::check_win_devel()           # check on Windows R-devel
devtools::check_rhub()                # cross-platform CRAN-like checks
covr::package_coverage()              # confirm coverage threshold

# 2. Update version + NEWS.md
usethis::use_version("minor")

# 3. Build + ship
devtools::build()                     # produces tar.gz in ../
devtools::release()                   # interactive CRAN submission helper
```

For internal releases (R-universe, PPM, internal mirror), skip
`devtools::release()` and use the org's publish flow — typically a tag-based
GitHub Action that runs `R CMD build` and uploads the artifact.

## Project Layout (non-package analyses)

For analysis projects that aren't installable packages, use the same
discipline minus the package machinery:

```
analysis-project/
  .Rproj
  DESCRIPTION              # Optional, but useful — captures deps
  renv.lock
  renv/
  R/                       # Reusable functions
  analysis/                # .Rmd / .qmd notebooks
  data-raw/                # Scripts that produce data/
  data/                    # Cleaned data outputs
  tests/testthat/          # Tests for R/
  README.md
```

The `DESCRIPTION` is optional here but recommended — it lets `devtools::test()`
and `renv` infer dependencies and gives a single place to record the
project's purpose and authors.

## Do / Don't

**Do:**

- Initialise `renv` on day one of every R project, package or not.
- Commit `renv.lock` on every dependency change.
- Declare runtime dependencies in `DESCRIPTION` `Imports:`, not `Depends:`.
- Use `usethis::use_*()` helpers — they encode the conventions for package
  scaffolding so you do not have to remember filenames and structure.
- Run `devtools::check()` locally before pushing — it is the same gate CI
  runs and catches the same problems faster.
- Pin GitHub installs to a tag or commit SHA; never to `main`/`master`.
- Maintain `NEWS.md` with a section per version.

**Don't:**

- Edit `NAMESPACE` by hand — it is regenerated from roxygen2 tags.
- Put runtime deps under `Suggests:` — they will not install on user
  machines.
- Use `Depends:` for ordinary package imports — that puts them on the
  user's search path and pollutes their session.
- Commit `renv/library/` — it is large and machine-specific.
- Skip `R CMD check` because tests passed. `check` catches a different
  class of problems (undeclared imports, broken examples, missing docs).
- Submit to CRAN without `devtools::check_win_devel()` — Windows-specific
  failures are the most common rejection reason.

## Common Pitfalls

1. **Forgetting `renv::snapshot()` after installing a new package.** The
   package works on your machine and breaks for everyone else because
   `renv.lock` was not updated. Make `snapshot()` part of your install
   muscle memory.

2. **`Depends:` instead of `Imports:`.** `Depends:` attaches the dependency
   to the user's search path on `library(yourpkg)`. This causes hard-to-
   diagnose conflicts. Use `Imports:` for almost everything.

3. **Hand-edited `NAMESPACE`.** roxygen2 regenerates it from `@export` and
   `@importFrom` tags. Hand edits get overwritten on the next
   `devtools::document()`. Add the missing tag in the source file.

4. **Using `library()` inside package code.** `R CMD check` rejects this.
   Inside a package, qualify with `pkg::fn` or add `@importFrom` to the
   relevant function.

5. **Pinning GitHub installs to a moving target.** `remotes::install_github("org/pkg")`
   pulls whatever is on `main` right now — different people get different
   builds. Always pin to `@<tag>` or `@<sha>`.

6. **Vignettes that fail on CI.** Vignette code runs during `R CMD check`.
   If it depends on a network or local file, the build breaks on the CI
   runner. Cache fixtures or wrap in `if (interactive())`.

7. **Skipping `check_win_devel()` before CRAN submission.** Windows-only
   failures (path separators, encoding, missing system tools) are the
   leading cause of CRAN rejections. Run the Windows R-devel check
   before every submission.

## Checklist

- [ ] `renv` initialised; `renv.lock`, `renv/activate.R`, `renv/settings.dcf` committed
- [ ] `DESCRIPTION` has accurate `Title`, `Description`, `Authors@R`, `License`, `URL`, `BugReports`
- [ ] `Depends: R (>= ...)` pins the minimum supported R version
- [ ] Runtime dependencies under `Imports:`, not `Depends:`
- [ ] `Suggests:` deps are guarded with `requireNamespace()` in `R/` code
- [ ] `Config/testthat/edition: 3` set when using testthat
- [ ] `NAMESPACE` regenerated via `devtools::document()` and committed
- [ ] `NEWS.md` updated for every version bump
- [ ] CI gate runs: `R CMD check --as-cran`, `lintr`, `styler --dry=fail`, `covr`
- [ ] `pkgdown` site built in CI and deployed on release tag
- [ ] GitHub installs pinned to tag or SHA, never to a branch name
- [ ] `devtools::check_win_devel()` clean before any CRAN submission
