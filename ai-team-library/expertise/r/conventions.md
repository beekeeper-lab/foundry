# R Stack Conventions

## Category
Languages

## Applies To

- data-scientist
- data-analyst
- data-engineer
- developer

These conventions are non-negotiable defaults for R projects on this team.
R has bifurcated into two dialects — modern tidyverse and base R — and the
defaults below codify the modern dialect. Deviations require an ADR with
justification. "I prefer it differently" is not justification.

---

## Defaults

| Concern              | Default Choice                          | Alternatives (ADR Required) |
|----------------------|-----------------------------------------|-----------------------------|
| R version            | 4.3+ (latest released minor, pin via `renv`) | LTS-only environments     |
| Style dialect        | tidyverse (dplyr / tidyr / ggplot2 / tibble / readr / purrr) | base R                |
| Modeling framework   | `tidymodels` (parsnip / recipes / workflows / yardstick) | `caret` (legacy)     |
| Plotting             | `ggplot2` (grammar of graphics)         | base graphics, `lattice`    |
| Dependency manager   | `renv`                                  | `packrat` (deprecated)      |
| Project structure    | RStudio Project (`.Rproj`) + `here` for paths | hand-rolled `setwd()`  |
| Formatter            | `styler`                                | manual / RStudio reformat   |
| Linter               | `lintr`                                 | none (not acceptable)       |
| Test framework       | `testthat` (3rd edition)                | `tinytest`, `RUnit`         |
| Documentation        | `roxygen2` (for packages)               | hand-written `.Rd` files    |
| Pipe operator        | Native `|>` (R 4.1+)                    | magrittr `%>%` (legacy code)|
| Large-data interop   | `arrow`, `duckdb`                       | bare `data.frame`           |

## 1. Project Structure

```
project-root/
  .Rproj                       # RStudio project file (anchors paths)
  DESCRIPTION                  # Package or analysis metadata
  NAMESPACE                    # Generated by roxygen2 (packages only)
  renv.lock                    # Dependency lockfile
  renv/                        # Local renv state (not committed beyond settings.dcf)
  .Rprofile                    # Project-local R options (sources renv/activate.R)
  R/                           # Function definitions (one logical group per file)
  tests/
    testthat/
      test-pricing.R           # Tests for R/pricing.R
      helper-fixtures.R        # Loaded automatically before tests
  data/                        # .rda objects (committed, small reference data)
  data-raw/                    # Scripts that build data/*.rda from sources
  inst/extdata/                # External files shipped with a package
  vignettes/                   # Long-form documentation (.Rmd)
  man/                         # Generated .Rd documentation (packages only)
  analysis/                    # Quarto / Rmd analysis notebooks (non-package)
  README.md
```

**Rules:**

- Use an RStudio Project (`.Rproj`) and the `here` package for paths. Never
  call `setwd()` inside scripts — paths must be relative to the project root.
- One package per repository. Multi-package monorepos are rare in R; if you
  need them, give each its own `DESCRIPTION` and `renv.lock`.
- Keep `R/` pure: function definitions only. Top-level executable code goes in
  `analysis/`, `data-raw/`, or `inst/scripts/`.
- Commit `renv.lock` for both packages and analyses. Lockfiles are how
  reproducibility happens in R.

## 2. Naming Conventions

| Element              | Convention             | Example                       |
|----------------------|------------------------|-------------------------------|
| Files in `R/`        | `kebab-case.R`         | `order-pricing.R`             |
| Functions            | `snake_case`           | `calculate_discount()`        |
| Variables            | `snake_case`           | `customer_orders`             |
| Constants            | `UPPER_SNAKE`          | `MAX_RETRY_COUNT`             |
| S3/S4/R6 classes     | `PascalCase`           | `OrderProcessor`              |
| Internal helpers     | `.leading_dot`         | `.validate_input()`           |
| Package data objects | `snake_case`           | `tax_rates`                   |

Do not use `.` as a word separator in function names (`my.function()`) — that
syntax conflicts with S3 method dispatch (`print.lm` is a method on class
`lm`). Use `snake_case` instead. The tidyverse style guide is the canonical
reference.

## 3. Formatting and Style

**Tools: `styler` (formatter) and `lintr` (linter).**

`styler` is the auto-formatter — analogous to `gofmt` or Ruff format. `lintr`
is the linter that gates CI. Keep them distinct: `styler` rewrites, `lintr`
reports.

```r
# Format a whole project
styler::style_pkg()
styler::style_dir("R/")
styler::style_file("R/pricing.R")

# Lint a project (fails CI on any infraction)
lintr::lint_package()
lintr::lint_dir("R/")
```

Project-local lintr config lives in `.lintr` at the project root:

```r
# .lintr
linters: linters_with_defaults(
  line_length_linter(100L),
  object_name_linter(styles = c("snake_case", "SNAKE_CASE")),
  cyclocomp_linter(complexity_limit = 15L)
)
exclusions: list("renv", "data-raw")
```

CI runs both: `styler` in `--dry=fail` mode (fail on unformatted code) and
`lintr` for static-analysis violations.

## 4. Tidyverse vs. base R

Tidyverse is the default. Base R is the documented alternative, allowed only
with an ADR.

**Use tidyverse when (default):**

- You are writing application code, analysis notebooks, modeling pipelines,
  or anything where readability and team velocity matter.
- The reader is likely to be another data scientist or analyst, not a
  package author or core-R maintainer.
- You want consistent function semantics across data manipulation
  (`dplyr`), reshaping (`tidyr`), iteration (`purrr`), and plotting
  (`ggplot2`).

**Use base R when (ADR required):**

- You are authoring a CRAN package and want zero non-base dependencies for
  install footprint or stability reasons.
- You are doing performance-critical work where dispatch overhead from
  tidyverse internals is measurable and matters.
- You are teaching or writing reference material that targets statisticians
  who think in base R idioms.

```r
# Tidyverse (default)
library(dplyr)

orders |>
  filter(status == "paid") |>
  group_by(customer_id) |>
  summarise(total = sum(amount), n = n()) |>
  arrange(desc(total))

# Base R (ADR-justified alternative)
paid <- orders[orders$status == "paid", ]
agg <- aggregate(amount ~ customer_id, data = paid,
                 FUN = function(x) c(total = sum(x), n = length(x)))
agg <- agg[order(-agg$amount[, "total"]), ]
```

**Pipe operator:** Use the native `|>` (R 4.1+). `magrittr`'s `%>%` is
legacy — keep it only when working with packages whose lazy-evaluation
semantics depend on it.

## 5. Function Design

- Every exported function takes a primary data argument first
  (`function(data, ...)`), so it composes with `|>`.
- Use explicit argument names at call sites for anything past position 2.
- Default to `tibble`s (not bare `data.frame`s) for return values.
- Validate arguments at entry. Use `stopifnot()` or `rlang::check_*()` for
  precondition checks. Never silently coerce.
- Return early on error conditions; avoid deeply nested `if`/`else`.

```r
calculate_discount <- function(data, tier = "standard", min_total = 50) {
  stopifnot(
    is.data.frame(data),
    "total" %in% names(data),
    is.numeric(data$total)
  )
  data |>
    mutate(
      discount = dplyr::case_when(
        total < min_total ~ 0,
        tier == "premium" ~ total * 0.15,
        TRUE ~ total * 0.05
      ),
      final_total = total - discount
    )
}
```

## 6. Documentation (roxygen2)

Every exported function in a package gets a `roxygen2` block. Roxygen
generates `man/*.Rd` and the `NAMESPACE` from comments above the function.

```r
#' Calculate per-row discount for an orders tibble.
#'
#' Applies a tier-based percentage discount to each order row, leaving the
#' input untouched and returning a new tibble with `discount` and
#' `final_total` columns added.
#'
#' @param data A tibble or data frame with at least a numeric `total` column.
#' @param tier One of `"standard"` or `"premium"`. Defaults to `"standard"`.
#' @param min_total Minimum order total below which no discount applies.
#'
#' @return A tibble with the original columns plus `discount` and `final_total`.
#' @export
#'
#' @examples
#' tibble::tibble(total = c(40, 100)) |> calculate_discount(tier = "premium")
calculate_discount <- function(data, tier = "standard", min_total = 50) {
  # ...
}
```

**Rules:**

- Every public function gets a `@param` line per argument and a `@return`
  line.
- `@export` exactly the functions that are part of the public API. Do not
  export internal helpers.
- Run `devtools::document()` after editing tags; commit the regenerated
  `NAMESPACE` and `man/`.

## 7. Error Handling

- Use `rlang::abort()` for programmer-facing errors with classed conditions
  so callers can `tryCatch()` selectively.
- Use `cli::cli_abort()` when the message benefits from `cli`'s formatting
  (interactive use, package-author UX).
- Reserve `warning()` for situations the user should know about but the
  function can still return a sensible value.
- Never use `stop()` with a bare string in new code — classed conditions are
  the modern idiom.

```r
if (any(data$total < 0)) {
  rlang::abort(
    "Negative totals are not allowed",
    class = "calculate_discount_error",
    body = c("i" = "Inspect rows where `total < 0` before calling this function.")
  )
}
```

## Do / Don't

**Do:**

- Use the tidyverse dialect for application and analysis code by default.
- Pin R version and dependencies with `renv` on every project.
- Use the native `|>` pipe (R 4.1+); reserve `%>%` for legacy interop.
- Run `styler` and `lintr` on every commit; CI must reject unformatted code.
- Write `roxygen2` documentation on every exported function.
- Use `here::here()` (or RStudio Projects) for paths; never `setwd()`.
- Validate function inputs at entry with `stopifnot()` or `rlang::check_*`.
- Return tibbles from functions that produce table-shaped data.

**Don't:**

- Use `.` as a word separator in function names — it conflicts with S3
  dispatch.
- Mix tidyverse and base-R idioms freely in the same pipeline; pick one
  per function.
- Call `library()` from inside a function or package — declare imports in
  `DESCRIPTION` and use `pkg::fn` qualifications.
- Use `T` and `F` for `TRUE` and `FALSE` — they can be reassigned and
  silently produce wrong answers.
- Rely on partial argument matching (`mtcars[, "mp"]` matching `mpg`).
  Always type the full name.
- Use `attach()` ever. It pollutes the global search path.

---

## Common Pitfalls

1. **`T` / `F` as Booleans.** `TRUE` and `FALSE` are reserved; `T` and `F`
   are ordinary symbols that happen to default to those values. A line like
   `T <- 0` somewhere upstream silently corrupts every later check. Always
   write `TRUE` / `FALSE`.

2. **Silent type coercion in `c()`.** `c(1, "a")` returns `c("1", "a")` —
   the numeric is coerced to character. If you need a heterogeneous
   container, use a `list()`.

3. **`subset()` and `transform()` in scripts.** Both use non-standard
   evaluation that breaks under indirection. Use `dplyr::filter()` and
   `dplyr::mutate()` in code; reserve `subset()`/`transform()` for
   interactive REPL use.

4. **Forgetting `drop = FALSE` on `[`.** `df[, "col"]` returns a vector,
   not a data frame, when you select a single column. Use
   `df[, "col", drop = FALSE]` or `dplyr::select(df, col)`.

5. **Using `setwd()` in scripts.** Hardcoded working directories break the
   moment another collaborator runs the code. Use RStudio Projects and
   `here::here()` instead.

6. **Calling `library()` inside package code.** Inside a package, you
   declare dependencies in `DESCRIPTION` and reference them with
   `pkg::function()` (or `@importFrom`). `library()` modifies the user's
   session and is forbidden by `R CMD check`.

7. **Returning the wrong shape under empty input.** `dplyr::summarise()` on
   an empty grouped tibble returns zero rows; downstream code that assumed
   one row per group breaks. Always test the empty-input case.

- [ ] Project uses an RStudio Project (`.Rproj`) and `here::here()` for paths
- [ ] `renv` initialised; `renv.lock` committed
- [ ] Tidyverse selected as the dialect (or ADR documents the base-R choice)
- [ ] `.lintr` config committed; `lintr::lint_package()` clean in CI
- [ ] `styler::style_pkg()` clean in CI (`--dry=fail`)
- [ ] `roxygen2` blocks on every exported function with `@param` / `@return`
- [ ] `NAMESPACE` regenerated via `devtools::document()` and committed
- [ ] No `setwd()`, `attach()`, `T`/`F`, or `library()` calls in package code

- [ ] Public functions take primary data first to compose with `|>`
- [ ] Functions validate inputs at entry; no silent coercion
- [ ] `pkg::fn` qualification used inside package code (or `@importFrom`)
