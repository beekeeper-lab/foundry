## Summary

Decision recorded as ADR-006 in `ai/context/decisions.md`. Foundry generates the AI team context; it does not generate stack-specific application code. The user initializes their application with a stack-appropriate command. The reversal direction — adding app scaffolding later — stays open; the irreversible direction — shipping scaffolders and then deprecating them — is what we avoid.

## Three-Stance Analysis

### Stance 1 — Planning-only (ACCEPTED)
Foundry scaffolds `.claude/`, `ai/`, `CLAUDE.md`, team composition, and the starter bean/charter. The user runs `npm create vite@latest`, `uv init`, `cargo new`, etc. to initialize their own app code.

- **Pros:** Small, focused surface. Dodges the stack-template treadmill (React/Vite/Next/Vue/Python/etc. each churn their generators independently, and Foundry would inherit every one of those churns). Aligns with BEAN-251's narrow `Edit(ai/**)` permission model. Matches the actual value proposition — Foundry's unique contribution is the AI team, not a "yet another starter template" generator. Cheapest to reverse later (add scaffolders on top of a small core).
- **Cons:** Day-1 gap: the generated project has no `package.json`/`pyproject.toml`/etc. Personas cannot "run the app" without a human step first. We mitigate this with a cheat sheet (`docs/starter-stacks.md`) naming the canonical init command per supported expertise.

### Stance 2 — Minimal stub per stack (REJECTED)
Emit a tiny stack-specific stub (e.g. `package.json` + `src/index.ts` for React/TS; `pyproject.toml` + `src/<name>/__init__.py` for Python).

- **Pros:** Closes the day-1 gap. Personas can run something immediately.
- **Cons:** Every stack needs its own generator; every generator needs to track upstream format changes (Vite 5 → 6, uv schema changes, `tsconfig.json` additions). A "minimal stub" is a lie — the minimum that actually works for React/TS today is ~8 files including a lockfile hint. Once shipped, users depend on it, and removing it later is a breaking change.

### Stance 3 — Delegate to ecosystem tools (REJECTED)
Shell out to `cookiecutter`, `npm create`, `yo`, etc. from within Foundry.

- **Pros:** Reuses well-maintained generators.
- **Cons:** Adds runtime dependencies Foundry does not otherwise need (Node, `cookiecutter`, `yo`). Composition spec would need to declare which ecosystem tool to invoke per expertise — a new configuration surface. Error paths multiply (what if `npm` isn't on PATH?). Still coupled to upstream template drift, just laundered through another tool.

## Reversibility Argument

The two directions are asymmetric:

- **Reversible:** Ship Planning-only today → add Stance 2/3 later if a compelling need emerges. The current generated output is a strict subset of any future expansion. No user's existing project breaks.
- **Irreversible:** Ship scaffolders today → later discover they can't keep up with upstream churn → deprecate them. Every generated project in the wild now has a `package.json` that Foundry no longer maintains. Users either keep the stale stub or migrate, both of which are support burden.

Choose the reversible direction.
