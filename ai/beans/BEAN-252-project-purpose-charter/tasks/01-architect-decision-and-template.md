## Goal

Pick between (a) scaffolding `ai/context/project-charter.md` and (b) promoting `project.description` to a required field with a min length. Record the decision (ADR or bean Notes) and define the file's structural skeleton.

## Inputs

- `bean.md` (this bean)
- `foundry_app/services/scaffold.py` (current scaffold pipeline)
- `foundry_app/core/models.py` `ProjectIdentity` (description is currently optional)

## Decision

**Choice: Option 1 — scaffold `ai/context/project-charter.md`.**

### Rationale

| Criterion | Charter file (chosen) | Required description |
|-----------|-----------------------|----------------------|
| Solves the audit complaint | Yes — discoverable, structured, hard to miss | Partial — a single-line description is still terse |
| Disrupts existing flows | No — additive only | Yes — wizard breaks for users mid-flight; CLI breaks YAML files in the wild |
| Forces day-1 thinking | Yes — TODO markers in 5 sections beg to be filled | Marginal — one line of text |
| Composes with future work | Yes — required-description can layer on top later | Locks in a stricter contract that's hard to reverse |
| Effort | Single file emission in scaffold pipeline + tests | Wider blast: model, validator, wizard, CLI, existing YAML migrations |

The charter file is **strictly additive**, mirrors the existing pattern of emitting `README.md` overlay-safely from `scaffold.py`, and gives personas a one-page source of truth. Promoting `description` to required is a bigger blast radius and could be revisited in a follow-up bean if the charter alone proves insufficient.

The decision is recorded as ADR-014 in `ai/context/decisions.md`.

## Charter Template Structure

The scaffolded file will follow this skeleton (BA defines the prose guidance per section in task 02):

```markdown
# Project Charter — <Project Name>

> **Status:** TODO — fill in this charter before the team begins substantive work.
> Personas should read this file first when opening the project.

<one-line description from spec.project.description, if present, else TODO line>

## Purpose

<TODO: one paragraph guidance>

## Audience

<TODO: one paragraph guidance>

## Success Criteria

- [ ] <TODO: criterion>
- [ ] <TODO: criterion>
- [ ] <TODO: criterion>

## Non-Goals

- <TODO: what this project explicitly does NOT do>

## Constraints

- <TODO: technical / organizational / regulatory constraints>

---
Generated by Foundry v<version> on <date>. Edit freely — this file is overlay-safe.
```

### Design points

1. **Filename:** `ai/context/project-charter.md` — sits next to `project.md` (the architecture/module map) and is the natural "what + who + done" companion.
2. **Overlay-safe:** Like README.md, only written if missing. Re-running scaffold over a populated charter must NOT clobber it.
3. **Header status line:** A `> **Status:** TODO` admonition at the top makes the unfilled state visible. The Team Lead can grep for `Status: TODO` to confirm the charter is filled.
4. **Description echo:** If `spec.project.description` is set, surface it under the header so the user sees their existing context inline. If absent, surface a TODO line so the absence is explicit.
5. **Footer:** Mirror the README footer (`Generated by Foundry vX on YYYY-MM-DD`) for traceability.
6. **No new dirs:** `ai/context/` already exists in the scaffold list; no scaffold structure change.
