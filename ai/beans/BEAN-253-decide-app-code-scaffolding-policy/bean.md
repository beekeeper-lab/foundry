## Problem Statement

External audit (2026-04-17): "No code scaffolding despite claiming React/TS. No `package.json`, `tsconfig.json`, `vite.config.ts`, `src/`, `.eslintrc`, no lockfile. A Developer cannot run anything on day 1."

This is a **design question about Foundry's scope**, not an implementation bug. Today, Foundry generates the *AI team scaffold* (`.claude/`, `ai/`, `CLAUDE.md`, agent/member files), not the *app code scaffold* (language-specific project skeleton). The audit is correct that this leaves the generated team with nothing to build on, but *adding app scaffolding* is a substantial scope expansion and deserves a conscious decision rather than a drive-by fix.

## Goal

Foundry's position on app-code scaffolding is documented and consistent: **Foundry generates the AI team context; the user initializes their own application code.** Generated projects say so explicitly.

**Decision (2026-04-17): Stance 1 — Planning-only.** Foundry will NOT scaffold stack-specific app code. Avoiding that scope keeps Foundry focused, dodges the stack-template treadmill (React/Vite/Vue/Next/Python/etc. version churn forever), and aligns with BEAN-251's permission model (`Edit(ai/**)` only).

The reversal direction — adding app scaffolding later — remains open. The irreversible direction — shipping scaffolders in the wild and then deprecating them — is what we avoid.

## Scope

### In Scope
- Record the Stance 1 decision as an ADR in `ai/context/decisions.md` with the full rationale (from the three-stance analysis).
- Update the generated project's `CLAUDE.md` template and `README.md` template to state the scaffolding policy clearly: the user initializes their app with stack-appropriate commands.
- Create `docs/starter-stacks.md` (project-level, not generated) as a cheat sheet of recommended init commands per supported expertise (Python, React/TS, Node, Rust, Go, etc.). Keep it short — one command per stack, link to authoritative docs.
- Cross-reference BEAN-251's Scope section: both beans produce consistent wording.

### Out of Scope
- Implementing per-stack scaffolders (explicitly rejected under Stance 1).
- Integrating with `cookiecutter`, `yeoman`, etc.
- Adding `spec.generation.scaffold_app_code` (rejected — a toggle implies both modes exist).
- Retrofitting existing generated projects.
